DeltaBot: Multi-Axis Coordinated Motion Control

7-Servo Delta Robot with Cartesian/Kinematic Coordinate Systems, Spiral Cam Profiles, and PackML-Structured Unit/Equipment Architecture on a Rockwell ControlLogix 1756-L81E Platform

Delta Kinematics Studio 5000 / 1756-L81E PackML (simplified)

Project scope. This project extends the single-axis simple motion control work into a full multi-axis coordinated platform: a Delta-style parallel robot with three joint axes, a rotary tool axis, and three virtual Cartesian axes tied to a CIP coordinate system. A 10-step procedural sequence drives MCLM coordinated linear moves through the Cartesian workspace, then calls a cam-profile calculator and launches two MATC time-driven cam moves to trace a parametric 8π-revolution spiral in X/Y while Z steps. Each axis also owns its own Equipment Module with a PackML-aware CM00 procedure for jog/stop/error handling.

1. System Architecture & Project Tree

The DeltaBot controller is a 1756-L81E ControlLogix running Studio 5000 Logix Designer. The project was structured using the ISA-88/PackML conventions already proven on the earlier Simple Motion project: a single unit (UN01_ExampleMachine) holds the procedural logic, while each physical axis is wrapped in its own Equipment Module (EM). All motion runs over EtherNet/IP CIP Motion with Time Sync, so every servo drive is clocked from the same grand-master on the backplane and coordinated linear moves execute on a common time base.

The coupling between the two coordinate systems is provided by UP02_Transform (delta forward/inverse kinematics). MCLM operates in the Cartesian frame so the programmer specifies moves in tool-tip space; the transform routine resolves the required joint angles in real time. The three virtual Cartesian members are read back through Cart_Coor.ActualPosition[5..7] (X, Y, Z respectively — indexed in the coordinate system's member order).

3. Unit Program — UP00_Procedure & Motion Math

UP00_Procedure is the master choreography of the robot: a 14-rung ladder that sequences ten steps using a set-reset state pattern. Step transitions key off either the .PC (process-complete) bit of the preceding MCLM, the .DN of a wait timer, or the .DNs of the two MCCP cam-profile calculations. The same motion-math block is repeated at every linear-move step — once per step, live at every scan — so the accel/decel are recomputed from the current robot position rather than assumed.

Move-math pattern (identical at each MCLM step)

distance[0] := ABS(Cart_Coor.ActualPosition[5] - moveN[0]);   // X error
distance[1] := ABS(Cart_Coor.ActualPosition[6] - moveN[1]);   // Y error
distance[2] := ABS(Cart_Coor.ActualPosition[7] - moveN[2]);   // Z error

accel       := (MoveSpeed ** 2)
             / (0.5 * (distance[0] + distance[1] + distance[2]) / 3);

The accel expression is a first-principles way of sizing the ramp: it picks the acceleration that exactly reaches MoveSpeed over half of the mean per-axis distance, so a short step uses a gentle ramp and a long step uses a stiff one. All moves then execute with:

4. Spiral Cam Profile — UP10_SpiralCamCalcPoints & UP03_SpiralCalcPts

Steps 7–8 draw a parametric 2-revolution spiral in the X/Y plane. Rather than stream a coordinated path, the program builds two cam profiles (one for X, one for Y) as a function of a common time-base master, then runs MATC on each virtual axis to play the cam back in time-driven mode. This is a classic trick when the workspace path is periodic or analytic: the PLC computes the geometry once and the motion planner interpolates the rest.

UP10 — build the cam points (Structured Text)

Figure 6. UP10_SpiralCamCalcPoints — 14-line structured-text routine that fills the Spiral_X_Cam and Spiral_Y_Cam CAM arrays with 101 points over 8π radians (two full spiral turns). Radius R = 2.0 grows linearly with θ, giving the classic Archimedean spiral r = Rθ.
NumOfPoints := 101;
R           := 2.0;

FOR Index := 0 TO NumOfPoints-1 DO
    Theta := Index * (8 * pi) / NumOfPoints;

    Spiral_X_Cam[Index].Master      := Index;
    Spiral_X_Cam[Index].Slave       := R * Theta * COS(Theta);
    Spiral_X_Cam[Index].SegmentType := 0;                 // linear

    Spiral_Y_Cam[Index].Master      := Index;
    Spiral_Y_Cam[Index].Slave       := R * Theta * SIN(Theta);
    Spiral_Y_Cam[Index].SegmentType := 0;
END_FOR;

UP03 — convert the point arrays into executable cam profiles

Figure 7. UP03_SpiralCalcPts — two parallel MCCP (Motion Calculate Cam Profile) blocks. Each takes the 101-point CAM array from UP10 and produces a CAM_PROFILE (Spiral_XCam_Pro, Spiral_YCam_Pro) that the motion planner can interpolate. The rung latches UP_Exec_StepDn_07 once both .DN bits are high.

Playback — per-axis MATC in the EM

Once the profiles exist, each Cartesian virtual axis calls MATC (Motion Axis Time Cam) to play its half of the spiral. The settings are identical for X and Y:

Figure 8. EM05_Axis_X / CM00_Procedure, rung 5 — MATC binds Axis_05_X to Spiral_XCam_Pro. Distance Scaling = 1.0 (full-size spiral), Time Scaling = 0.15 (plays the cam in 15 % of master time — effectively ≈ 7× faster than 1 s/point), Execution Mode = Once, Schedule = Immediate, Instruction Mode = Time Driven Mode.

Because MATC is time-driven, the spiral traces itself autonomously on each axis — the virtual X and Y axes share nothing but the common time base, yet produce a smooth 2D path because both cams were built from the same Theta sweep. Meanwhile Axis_04_R runs a short jog sequence in its own EM so the tool spins while the spiral draws.

5. Equipment Modules — Per-Axis CM00_Procedure

Each axis EM owns an identical CM00 pattern — 8 or 9 rungs of PackML-aware logic that map the unit step flags into motion instructions on that axis:

  • Rung 0: placeholder NOP (header).
  • Rung 1: on s:fs (first scan), COP the default MI_Init struct into the working motion-instruction tag.
  • Rung 2: on first scan, UP_Resetting or UP_Idle, unlatch every step bit.
  • Rung 3: if no step is active, latch No_Steps_Active.
  • Rung 4: if UP_Execute and No_Steps_Active, seed Step_1.
  • Rungs 5–7: the axis's actual motion instructions (the differ per EM — see below).
  • Last rung: AFI-guarded error latch to EM_ProcedureError[EM_Number].

EM03_Axis_R — jog during the spiral (MAJ + MAS)

Figure 10. Rung 6 of the same routine — an MAS (Motion Axis Stop) kills the jog at UP_Exec_Step_09, changing decel but preserving the decel-jerk, so the tool stops smoothly before the final return move.

7. Tuning & Trend Analysis (Weeks 2 – 3)

Three trend experiments were run to visualise how accel/decel jerk shapes the Cartesian path during the 6-step linear sequence (before the spiral). Cart_Coor.ActualPosition[0..2] (X/Y/Z — at coordinate-system indices 0/1/2) are plotted against time on the same chart; a companion chart plots X vs. Z to show the projected path.

Jerk = 100 % of time (trapezoidal — fastest cycle)

Figure 12. X/Y/Z position vs time with jerk = 100 %. Total cycle ≈ 45 s. The ramps are sharp-elbowed — jerk is effectively infinite at the transition, which is mechanically the harshest but temporally the fastest.
Figure 13. X vs Z projection, jerk 100 %. The motion is dominated by straight legs (pure trapezoidal), giving a crisp polygonal path.

Jerk = 50 % of time (half-S, half-trap)

Figure 14. Jerk = 50 %. The shoulders of each ramp now round slightly; cycle lengthens a few seconds but peak jerk is halved.
Figure 15. X vs Z, jerk 50 %. Corners of the polygonal path stay tight because termination type is still actual tolerance — the motion planner fully completes each step before the next launches.

Jerk = 10 % of time (full S-curve)

Figure 16. Jerk = 10 %. Ramps are now deeply rounded (essentially an S-curve), cycle time grows noticeably, but mechanical stress on the delta parallelograms drops dramatically.
Figure 17. X vs Z at jerk 10 %. The projected path is still linear segment-by-segment because each step terminates individually, but intra-segment it is gentler.

Week-3 run — with the spiral cam playing

Figure 18. Full week-3 run: linear steps 1–6, then the spiral's time-driven MATC on X/Y (the two ramped-sinusoid humps on the X and Y channels around the midpoint), then linear steps 9–10 to return home.
Figure 19. X vs Y during the linear portion (the trend chart clipped out the spiral curvature at this zoom). The diagonal trace corresponds to the (50, 50) → (−50, −50) diagonal traversed across steps 1, 3, 4, 5, 6.
Tuning takeaway. Jerk percentage trades cycle time for mechanical friendliness on an almost linear basis for this workload — 10 % jerk is ~1.5× the total time of 100 % jerk but is what you would run in production on a delta robot, because the parallel kinematics amplifies shock at the tool through all three J-axes simultaneously.

8. Engineering Takeaways

  • Coordinate systems over per-axis moves. Programming moves in Cart_Coor (with UP02_Transform doing the delta inverse kinematics under the hood) meant the unit procedure contains pure Cartesian geometry — easier to review, simulate, and tune than seven independently choreographed joint motions.
  • Cam profiles for analytic paths. The 2-revolution spiral could have been streamed point-by-point, but building it once (UP10) → converting it to a CAM_PROFILE (UP03) → playing it back with MATC at Time Scaling = 0.15 is vastly smoother and cheaper per scan.
  • Live-computed accel. Computing accel = MoveSpeed² / (½·mean-distance) at every step makes the procedure position-robust: homing offsets or operator jogs don't break the ramp.
  • PackML in every module. The UP_Idle / UP_Execute / UP_Resetting flags appear in every EM's CM00, so each axis independently agrees with the unit on when to start, stop, or clear steps. This is what lets the R-axis jog and the X/Y cams fire in parallel without cross-contamination.
  • Jerk is the production knob. The three trend sets (100 %, 50 %, 10 %) show jerk is the first-order lever for mechanical stress on a delta — more useful to vary than speed itself.
